Step 1: Build images (30 seconds)
$ cd /home/amin/Desktop/vm-migrator
$ ./docker/scripts/build-offline.sh --ver v1.0

Step 2: Configure (2 minutes)
$ vim .env
  ├─ Set DB_PASSWORD (change from 'admin')
  ├─ Set DB_ROOT_PASSWORD (change from 'rootpassword')
  └─ Verify VMWARE_ESXI_CONVERSION_TRANSPORT=vddk

Step 3: Start services (1 minute)
$ docker-compose -f docker-compose.offline.yml up -d

Step 4: Initialize database (30 seconds)
$ docker-compose -f docker-compose.offline.yml exec backend \
  python manage.py migrate --noinput

Step 5: Access application
Frontend:  http://localhost:3000/
Admin:     http://localhost:8000/admin/
API:       http://localhost:8000/api/

1. VDDK PLUGIN:
   ├─ Dockerfile includes VDDK libraries and headers
   ├─ nbdkit VDDK plugin may not be available in system packages
   ├─ If VDDK plugin missing, use libvirt_esx transport instead
   └─ Conversion will still work, just without direct VDDK plugin access

2. HOST KERNEL MOUNTS:
   ├─ Conversion worker mounts /boot:/boot:ro and /lib/modules:/lib/modules:ro
   ├─ Required for libguestfs supermin kernel access
   ├─ Must have root permissions to read these directories
   └─ Already configured in docker-compose.offline.yml

3. CELERY WORKER:
   ├─ Runs as root (necessary for kernel access)
   ├─ Has access to VDDK at /opt/vmware-vddk
   └─ Preflight checks verify VDDK availability on startup

4. MARIADB & REDIS:
   ├─ Still pulled from Docker registries (only internet access point)
   ├─ To make fully offline, pre-pull on isolated machine first:
   │  $ docker pull mariadb:10.11.8
   │  $ docker pull redis:7.2.5-alpine
   └─ Images then cached locally, no further registry access needed

5. VERSION TAGGING:
   ├─ Always tag images with versions: vm-migrator/backend:v1.0
   ├─ Helps track which configuration/dependencies used
   └─ Use semantic versioning for clarity
